## 什么是 Harness-Pro？（5分钟）

### 一句话解释

想象你是一个项目经理，手下有几位程序员。你不会自己去写代码，而是：
1. 把大任务拆成小任务
2. 分配给程序员
3. 检查他们的工作
4. 确保最终成果合并到主项目

**Harness-Pro 就是这套工作流程的自动化版本。**

### 解决什么问题？

| 传统开发问题 | Harness-Pro 的解决方案 |
|-------------|------------------------|
| 一个大分支，几千行代码，评审痛苦 | 每次只改 3-4 个文件 |
| 多人同时改同一目录，容易冲突 | 每人有独立工作目录 |
| 代码写完了才发现测试不足 | 每个小任务都必须有测试 |
| 项目进度不透明 | 每个任务状态清晰可见 |
| 代码回滚困难 | 小任务独立回滚很容易 |

### 它的本质

```
Harness-Pro = 工程协作方法论 + 执行规范
           （不是产品代码，是工作方式）
```

## 核心概念：用生活场景理解

### 概念 1：Main Agent vs Worker Agent

**生活比喻：餐厅**

```
┌─────────────────────────────────────────┐
│         Main Agent（餐厅经理）           │
│                                         │
│  职责：                                  │
│  • 接收顾客订单（理解用户需求）          │
│  • 拆分任务（炒菜、配菜、摆盘）          │
│  • 分配给厨师（调度 Workers）            │
│  • 检查菜品质量（评审结果）              │
│  • 决定是否上菜（批准合并）              │
│                                         │
│  不做的事：                               │
│  ✗ 不亲自炒菜（不写代码）                │
│  ✗ 不替厨师处理原料（不修改 Worker 代码） │
└─────────────────────────────────────────┘

         ↓ 分配任务

┌─────────────────────────────────────────┐
│       Worker Agents（厨师们）            │
│                                         │
│  职责：                                  │
│  • 按照菜单做菜（执行任务文档）          │
│  • 品尝味道（运行测试）                  │
│  • 修改味道不足的地方（处理评审反馈）      │
│  • 完成后装盘（提交合并）                │
│                                         │
│  不做的事：                               │
│  ✗ 不自己设计菜单（不做任务规划）        │
│  ✗ 不随意加菜（不扩大任务范围）          │
└─────────────────────────────────────────┘
```

### 概念 3：3-4 文件规则

**为什么是 3-4 个文件？**

```
想象你在一篇论文上签字：

【大分支方案】
论文有 100 页，一次性全看完再签字
→ 很难看出问题
→ 发现问题不知道在哪一页
→ 签错字后果严重

【小任务方案】
每次只看 1-2 页
→ 快速理解内容
→ 问题一目了然
→ 签错只影响这一页
```

**3-4 个文件典型组成：**

```
任务：添加用户模型
文件清单（共 3 个）：
├── user.py                 (1) 用户模型代码
├── tests/test_user.py      (2) 用户测试代码
└── README.md               (3) 文档更新（可选）

任务：添加用户存储
文件清单（共 3 个）：
├── storage.py             (1) 存储代码
├── tests/test_storage.py   (2) 存储测试
└── user.py                (3) 修改相关导入（可选）
```

### 概念 4：Git Worktree = 独立厨房

**传统方式 vs Worktree 方式**

```
传统方式（共享厨房）：
厨师 A、B、C 都在同一个厨房
→ A 在炒菜，B 想用同一个锅，冲突！
→ C 打翻调料，所有人都受影响
→ 很难同时做几道菜

Worktree 方式（独立厨房）：
每个厨师有独立的厨房
├── 厨房 A：.worktrees/task-001/
├── 厨房 B：.worktrees/task-002/
└── 厨房 C：.worktrees/task-003/

→ 互不干扰，可以同时做菜
→ A 的调料洒了，不影响 B、C
→ 做完后，把菜端到主厨房
```

### 概念 5：固定团队 = 常驻员工

**错误做法：临时工模式**

```
任务 001: 招个临时工小明，做完就辞退
任务 002: 招个临时工小红，做完就辞退
任务 003: 招个临时工小刚，做完就辞退

问题：
• 每次都要重新培训
• 不熟悉项目风格
• 沟通成本高
```

**正确做法：固定团队**

```
一开始就招 3 个员工：小明、小红、小刚

任务 001 → 小明
任务 002 → 小红
任务 003 → 小刚
任务 004 → 小明（又轮到他了）
...

好处：
• 熟悉项目
• 效率越来越高
• 节省培训成本
```

## 实战案例：任务管理器开发

### 案例背景

我们要开发一个任务管理系统，功能包括：
- 添加任务
- 列出任务
- 标记完成/未完成
- 保存到文件

### 案例：如何拆分任务

#### 错误的拆分方式

```
任务 1：整个任务管理系统
  ├── user.py (200 行)
  ├── storage.py (300 行)
  ├── api.py (400 行)
  ├── tests/test_all.py (500 行)
  └── ...

问题：一个任务 1400+ 行代码
      • 难以评审
      • 难以调试
      • 出问题难定位
```

#### 正确的拆分方式

```
任务 001：基础任务模型（3 个文件）
  ├── task.py (50 行)
  └── tests/test_task.py (100 行)

任务 002：任务存储（3 个文件）
  ├── storage.py (80 行)
  ├── tests/test_storage.py (150 行)
  └── task.py (修改导入)

任务 003：完成状态（2 个文件）
  ├── task.py (添加方法)
  └── tests/test_completion.py (80 行)

任务 004：文件持久化（3 个文件）
  ├── persistence.py (60 行)
  ├── tests/test_persistence.py (100 行)
  └── storage.py (集成)

每个任务都小而聚焦！
```

### 案例：任务文档示例

#### Task 001 完整文档

```markdown

TASK-001

## Goal
创建 Task 类，表示单个任务

## Task Type

## Scope

  - mark_complete()

创建 tests/test_task.py，包含：
- 基本创建测试
- 类型验证测试
- 完成状态测试
- 边界情况测试

## Non-Goals
- 不涉及存储（Task 002）
- 不涉及持久化（Task 004）
- 不涉及重复验证（Task 005）

## Repository Context
- Repository: /path/to/project
- Base branch: main
- Depends on: None

## Branch Name
feature/task-001-model

## Worktree Path
.worktrees/task-001-model

## Required Agent
harness-pro-worker

## Expected File Budget
2 files: src/task.py, tests/test_task.py

## Dependencies
None

## Validation Commands
```bash
pytest tests/test_task.py -v
python -c "from src.task import Task; t = Task('Test'); print(t)"
```

## Acceptance Criteria
- [ ] Task 类存在且构造正确
- [ ] title 非字符串时抛出 TypeError
- [ ] 三种状态方法工作正常
- [ ] 测试覆盖率 > 90%
- [ ] 所有测试通过
```

## 常见问题解答

### Q1：为什么要 3-4 个文件？不能多一点吗？

**答：当然可以，但要谨慎。**

```
✓ 3-4 个文件：最优，快速评审
✓ 5-6 个文件：可接受，如果紧密相关
✗ 10+ 个文件：太大，建议拆分
```

**什么时候可以多一点？**

```
# 紧密相关的代码，可以多一点
任务：添加用户认证
├── auth.py           (认证逻辑)
├── user.py           (修改用户模型)
├── middleware.py     (中间件)
├── tests/test_auth.py (测试)
└── tests/test_user.py (相关测试)
共 5 个文件，可以接受

# 不相关的代码，必须拆分
任务：不要这样做
├── user.py          (用户功能)
├── product.py       (产品功能)  ← 不相关！
├── order.py         (订单功能)  ← 不相关！
└── ...
应该拆成 3 个独立任务
```

### Q2：如果任务实际做的时候发现需要更多文件怎么办？

**答：立即报告，不要自行扩展。**

```
情况：任务文档说 3 个文件
实际做的时候：发现需要 5 个文件

✗ 错误做法：
默默做完 5 个文件
提交评审
→ Main Agent 可能会拒绝

✓ 正确做法：
1. 停下来
2. 报告："任务实际需要 5 个文件，因为..."
3. 等待 Main Agent 决定：
   - 拆分成两个任务？
   - 批准扩到 5 个文件？
   - 缩小范围？
```

### Q3：Main Agent 能否直接修改代码？

**答：不能，这是核心规则。**

```
Main Agent 禁止做的事：
✗ 直接修改 worker 代码
✗ 帮 worker 合并
✗ 替 worker 写测试
✗ 修复 worker 的 bug

Main Agent 应该做的事：
✓ 写任务文档
✓ 评审 worker 提交的代码
✓ 批准或拒绝
✓ 如果有 bug，创建修复任务让 worker 做
```

**为什么？**

```
如果 Main Agent 直接改代码：
1. 职责模糊，不知道谁负责什么
2. 失去隔离的好处
3. 评审变得没有意义
```

### Q4：两个任务能否并行？

**答：可以，但必须没有依赖。**

```
可以并行的情况：

TASK-A（添加用户模型）
TASK-B（添加产品模型）

→ 没有依赖，可以同时做

不能并行的情况：

TASK-001（创建任务模型）
TASK-002（创建任务存储）

→ 002 依赖 001，必须先做完 001
```

### Q5：Worker 完成任务后，worktree 怎么办？

**答：保留一段时间后清理。**

```
刚完成：保留
原因：可能需要回滚或重新检查

一段时间后（比如 1 周）：可以清理
命令：git worktree remove .worktrees/task-xxx

重要分支：可以保留
原因：需要参考或学习
```

### Q6：如何追踪任务进度？

**答：使用任务图。**

```
任务状态追踪：

TASK-001: [Merged] ✓
TASK-002: [Under Review] 🔄
TASK-003: [Running] 🏃
TASK-004: [Ready] ⏳
TASK-005: [Pending] 📝

图例：
✓ Merged - 已完成合并
🔄 Under Review - 等待评审
🏃 Running - Worker 正在做
⏳ Ready - 可以分配
📝 Pending - 等待依赖
```

### Q7：如果测试失败了怎么办？

**答：Worker 必须修复，不能提交。**

```
Worker 的责任：

1. 实现代码
2. 添加测试
3. 运行测试

如果测试失败：
✗ 错误："代码写完了，测试下次再说"
✓ 正确：
   - 修复代码
   - 或修复测试（如果测试写错了）
   - 确保所有测试通过
   - 然后才提交评审
```

### Q8：Main Agent 和 Worker 必须是 AI 吗？

**答：不完全是，这是角色定义。**

```
Main Agent 可以是：
- AI（Claude）
- 人类项目经理
- 脚本/工具

Worker Agent 可以是：
- AI（Claude）
- 人类程序员
- 自动化工具

重要的是角色的职责，不是谁扮演
```

## 进阶技巧

### 技巧 1：快速识别并行任务

```
问题：如何快速找出可以并行的任务？

方法：绘制依赖图

任务依赖：
001 → 002 → 004
   → 003 → 004

可视化：
      001
      /  \
    002  003
      \  /
       004

可以并行的：
• 002 和 003 可以同时做

不能并行的：
• 001 必须先做
• 004 必须等 002 和 003 都做完
```

### 技巧 2：优先级排序

```
当多个任务都 Ready 时，选择哪个？

考虑因素：

1. 依赖解锁
   做了这个任务，能解锁最多后续任务
   → 优先级高

2. 风险和复杂度
   复杂的任务先做
   → 问题早发现，风险可控

3. 用户价值
   最核心的功能先做
   → 尽早交付价值

示例：
TASK-A: 核心数据模型（解锁 5 个任务）
TASK-B: UI 美化（不解锁任何任务）

→ 优先做 TASK-A
```

### 技巧 3：Worker 空闲时做什么？

```
没有实现任务时，Worker 不要闲着

可以做的任务：
• 回归测试：验证最近合并的代码
• Bug 修复：修复发现的小问题
• 文档更新：补充 API 文档
• 性能测试：运行性能测试
• 代码清理：清理已知的技术债

原则：只要有用且安全，就分配给 Worker
```

### 技巧 4：任务文档的迭代

```
第 1 版：刚开始做的时候
很简略，描述基本需求

第 2 版：做完第一个任务后
根据经验更新模板，补充细节

第 3 版：项目进行中
持续优化，积累最佳实践

最终：形成团队标准的任务文档模板
```

### 技巧 5：处理评审反馈

```
评审后收到反馈，怎么办？

✓ Worker 的责任：
   - 在同一个 worktree 工作
   - 根据反馈修改
   - 重新运行测试
   - 再次提交评审

✗ 不要做的事：
   - 创建新的 worktree
   - 忽略反馈
   - 期望 Main Agent 修复

目标：直到评审通过
```

## 总结

### 核心要点回顾

```
1. 职责分离
   Main Agent = 编排者，不写代码
   Worker Agent = 执行者，不规划

2. 小任务
   每次改 3-4 个文件
   易评审，易回滚

3. 任务文档
   明确目标、范围、验收标准
   是执行的契约

4. 隔离执行
   Git Worktree 提供独立环境
   不互相干扰

5. 固定团队
   创建一次，复用整个流程
   节省成本，提效率
```
