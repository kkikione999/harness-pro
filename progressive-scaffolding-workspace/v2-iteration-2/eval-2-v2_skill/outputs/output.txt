SECTION 2: CAN AN AGENT SELF-DEBUG A RUNTIME ERROR?
================================================================================

**Answer: NO — not without human help, and not even close.**

The core blocker is architectural: this is a read-only research project with no build system. But even setting that aside (hypothetically assuming the project could run), the scaffolding is fundamentally insufficient for self-debugging. Here is why, broken down by the O→D→A→V loop:

### OBSERVE (O) — Fails completely

An agent debugging a runtime error needs to:
  1. Read structured logs to identify the error and its context
  2. Trace the correlation ID through all components
  3. Query internal state to understand what went wrong

What actually happens:
  - The app emits raw console text to stdout — no JSON, no level/timestamp/metadata fields
  - An agent sees a wall of text and must use pattern matching to infer structure
  - No correlation ID is injected into any log line
  - Internal state (AppState) is opaque — the agent cannot query it

**Behavioral consequence**: An agent can read raw error messages (O1 partial), but cannot systematically trace cause → effect across components, cannot correlate log entries to a specific request, and cannot determine what internal state caused the failure.

### DECIDE (D) — No control surface exists

An agent needs a decision map:
  - Config issue   → setHarnessOverride()
  - Missing data   → injectState()
  - Feature issue  → setFeatureFlag()
  - Need to verify → harnessLog() + re-trigger

What actually exists:
  - Zero control functions are exported from any module
  - The agent cannot change behavior without modifying source files manually
  - Manual file edits require the agent to know what to change AND have a theory of what is broken
  - No feature flag toggle means the agent cannot test code paths in isolation

**Behavioral consequence**: An agent debugging a config-related error (e.g., wrong API base URL) cannot override the config at runtime. It must edit a config file, restart, and re-run — a multi-step process requiring human-level understanding of the dependency graph.

### ACT (A) — Cannot execute control functions

An agent needs to call harness functions:
  - `node -e "require('./src/config').setHarnessOverride(...)"`
  - `node -e "require('./src/state').injectState(...)"`

What actually happens:
  - No such functions exist — the agent gets a module-not-found or undefined error
  - Even if functions existed, `HARNESS_CONTROL=1` env flag is never checked in the TypeScript source
  - The settings system (`src/utils/settings/`) has complex merge logic (lodash mergeWith, 5 scopes) but no single override function

**Behavioral consequence**: The agent's toolkit is limited to file reads and tool calls. For self-debug, it must either guess the fix and write a file, or ask a human.

### VERIFY (V) — No behavioral confirmation

An agent needs to confirm the fix worked:
  - Re-parse structured logs for a behavioral delta
  - Confirm the error no longer appears
  - Confirm the correct code path was taken

What actually exists:
  - No structured logs to re-parse
  - No internal state API to query for confirmation
  - The agent must infer success from opaque terminal output (Ink renders to ANSI text)

**Behavioral consequence**: An agent cannot close the loop. It does not know if its fix actually worked without human confirmation.

SECTION 3: BEHAVIORAL GAP ANALYSIS — WHY EACH GAP MATTERS
================================================================================

### Controllability Gaps

**C1=0: No runtime config control**
  Why it matters for autonomy: The most common class of self-debug tasks involve changing a configuration value (API key, timeout, feature flag). Without runtime config mutation, every such fix requires: (1) identify config file, (2) edit it, (3) restart the app, (4) verify. Steps 1-4 each introduce a failure point where the agent needs deep project knowledge. A `setHarnessOverride()` collapses this to one atomic call.

**C2=0: No feature flag control**
  Why it matters: In a codebase of 432k lines, different code paths are guarded by feature flags. An agent debugging a behavioral difference (e.g., "logs differently on macOS vs Linux") cannot toggle the relevant flag — it must understand the flag's existence from source and manually edit. Feature flags are designed to decouple deploy from enable — without an agent-accessible toggle, this decoupling is lost.

**C3=0: No state injection**
  Why it matters: Many bugs require injecting test data to reproduce. Example: "API returns 404 for user ID 42." Without `injectState()`, the agent cannot simulate this user. It must set up a real database state, which may be impossible in a sandboxed agent environment. State injection collapses "reproduce bug" from an infrastructure problem into a function call.

**C4=0: No middleware injection**
  Why it matters: MidDLEWARE enables cross-cutting behavioral observation (timing, request/response logging, error interception). Without it, each observation point must be hand-coded by the agent. For a 1,900-file codebase, finding where to add observation is non-trivial. A middleware pipeline that the agent can inject into provides a universal interception point.

### Observability Gaps

**O1=0: No semantic logs**
  Why it matters: This is the foundational gap. Without structured JSON logs, every log-reading task is a parsing heuristic that will break. Agents must write regex patterns to extract fields from freeform text — fragile, brittle, and project-specific. `harnessLog()` with a defined schema (level, timestamp, correlationId, message, metadata) makes log parsing a deterministic extraction problem.

**O2=0: No correlation propagation**
  Why it matters: In a multi-component system (REPL → QueryEngine → query.ts → tools), a single user action generates logs across dozens of files. Without a correlation ID, an agent debugging a multi-step failure cannot connect log entries to the same request. The agent sees N log files and must guess which entries belong together. Correlation propagation is what makes distributed tracing tractable.

**O3=0: No internal state query**
  Why it matters: AppState contains critical runtime information: current conversation ID, active tools, permission state, MCP connections. When a tool fails, knowing the exact AppState at failure time is diagnostic gold. Without `queryState()`, the agent must infer state from side effects (what the terminal rendered) rather than querying it directly.

**O4=0: No event visibility**
  Why it matters: Internal events (tool_dispatched, api_request_sent, state_changed) are the "why" behind observable outputs. Without event emission, the agent only sees the final output (tool result XML) — it cannot see the intermediate causal chain. Event visibility is what separates "error message" from "root cause."

SECTION 4: EXISTING SCAFFOLDING — WHAT IS PRESENT AND WHAT IT PROVIDES
================================================================================

The project has a .harness/ directory with v1-style surface wrappers:

### Controllability Scripts (.harness/controllability/)
  - `start.sh` — detects project type (npm/go/make) and starts the service; generates CORRELATION_ID in shell
  - `stop.sh` — kills process by PID file or name
  - `verify.sh` — checks PID file + optional HTTP health endpoint
  - `test-auto.sh` — framework detection (npm/go/make); falls through to "no test framework" exit
  - `analyze-deps.sh` — DFS-based dependency cycle detection across src/ TypeScript files
  - `lint-import-direction.sh` — enforces layer hierarchy (utils ← services ← tools)
  - `Makefile` — orchestrates the above with make targets

**Assessment**: These are all passive wrappers. They do NOT inject behavioral control into the application. They check file existence, process status, and import patterns. The CORRELATION_ID is set in the shell but never read by the TypeScript code.

### Observability Scripts (.harness/observability/)
  - `log.sh` — queries .log files in .harness/observability/logs/ (recent/search/follow)
  - `trace.sh` — wraps a command with CORRELATION_ID injection in the shell
  - `health.sh` — JSON health status (PID + optional HTTP health endpoint)

**Assessment**: `log.sh` reads files that may not exist (logs directory is empty, app never starts). `trace.sh` sets a shell variable that the TypeScript pipeline ignores. These provide scaffolding infrastructure but no runtime observability.

### CI Pipeline (.harness/ci/ci-pipeline.yml)
  - GitHub Actions: lint → dependency-check → health → verify → test-auto
  - Targets: lint-import-direction, cycle detection, health check, verify, test-auto

**Assessment**: The CI pipeline is well-structured for a standard project, but the test-auto step falls through to "no test framework" for Open-ClaudeCode, making the verify + test-auto jobs non-functional.

### What the .harness scaffolding CAN do:
  1. Generate correlation IDs in shell processes
  2. Run lint checks on import direction
  3. Detect dependency cycles
  4. Check process health (PID-based)
  5. Query log files (if they exist)

### What the .harness scaffolding CANNOT do:
  1. Modify TypeScript runtime behavior
  2. Emit structured logs from within the app
  3. Expose config/state APIs to the agent
  4. Propagate correlation IDs through the query pipeline
  5. Enable self-debug without human intervention

SECTION 5: INJECTION POINT ANALYSIS
================================================================================

For a Tier 1-2 progressive scaffolding intervention, the following injection
points are identified:

### Config Injection Point (C1)
  Primary: `src/utils/settings/types.ts` (SettingsSchema Zod definitions)
  Secondary: `src/utils/settings/internalWrites.ts` (settings merge logic)
  Pattern idiom: Zod schemas with `.optional().describe()` — already used
  Agent usage: `setHarnessOverride()` would read from `_harnessOverrides` map
    before falling through to the existing 5-scope merge system
  Risk: Low — settings are already centralized in this module

### Log Injection Point (O1)
  Primary: `src/ink/output.ts` (Ink terminal output renderer)
  Secondary: `src/utils/bash/logger.ts` (if it exists — console.log wrappers)
  Pattern idiom: Ink uses `boxen` + ANSI rendering — no existing structured logging
  Agent usage: `harnessLog()` would call both `console.log()` and write JSON to stdout
  Risk: Medium — Ink output is performance-critical; JSON stdout may interfere
    with terminal rendering. Solution: write JSON to stderr or a pipe.

### Feature Flag Injection Point (C2)
  Primary: `src/utils/settings/types.ts` (settings schema has experimental flags)
  Secondary: `src/utils/bash/featureFlags.ts` (feature() function from bun:bundle)
  Pattern idiom: `feature('FLAG_NAME')` checked via bun:bundle at bundle time
  Agent usage: `setFeatureFlag()` would overlay on top of feature() result
  Risk: High — feature() is resolved at bundle time; a runtime overlay is a
    behavioral change from the original CLI. Acceptable for harness use only
    when HARNESS_CONTROL=1.

### State Injection Point (C3/O3)
  Primary: `src/state/AppStateStore.ts` (Zustand-like store)
  Secondary: `src/state/store.ts` (store initialization)
  Pattern idiom: `getState()` / `setState()` + selector pattern — already reactive
  Agent usage: `injectState()` / `queryState()` would read/write the store directly
  Risk: Medium — AppState is central to all UI. Direct mutation bypasses the
    intended immutable update pattern. Should use `setState()` internally.

### Correlation ID Injection Point (O2)
  Primary: `src/query.ts` (single API turn — the narrowest point in the pipeline)
  Secondary: `src/QueryEngine.ts` (orchestrator)
  Pattern idiom: pipeline functions receive context object — correlationId can
    be added to context without changing signatures if added to a context bag
  Agent usage: `HARNESS_CORRELATION_ID` env var read at query.ts entry
  Risk: Low — purely additive; does not change existing behavior

### Middleware Injection Point (C4)
  Primary: `src/Tool.ts` (buildTool() factory) or `src/tools.ts` (tool registry)
  Secondary: Each tool's `call()` entry point
  Pattern idiom: Each tool implements `call(args, context)` — a middleware
    registry would wrap these calls with pre/post hooks
  Agent usage: `injectMiddleware()` would add to a registry read by buildTool()
  Risk: High — Tool dispatch is synchronous; adding middleware requires
    modifying every tool call site or using a Proxy/dynamic wrapper.

SECTION 6: RECOMMENDED STARTING TIER
================================================================================

Recommended: Tier 2 — Observability (not Tier 1 Config Control)

Rationale:
  Tier 1 (Config Control, ~20 lines) would be immediately useful IF the app
  could run, but the app cannot run (no build system). Adding C1/C2 is premature
  when the app cannot be started to exercise those functions.

  Tier 2 (Observability, ~30 lines) should come first because:
    (a) The app CAN be read — source analysis is always available
    (b) Structured logs enable the O→D loop even without ACT capability
    (c) Semantic logs + correlation propagation are the minimum bar for
        an agent to OBSERVE and DECIDE in a self-debug scenario
    (d) An agent with O1+O2 can at least observe and diagnose; it cannot
        act, but the diagnostic output is still valuable
    (e) Type B injection (structured log) at `src/ink/output.ts` or a new
        `src/utils/harness.ts` is low-risk: JSON to stderr preserves
        terminal rendering while making logs parseable

  Only after O1+O2 is working should Tier 1 (C1) be added, followed by
  Tier 3 (C2) and Tier 4 (C3/O3).

SECTION 7: SUMMARY
================================================================================

Can an agent self-debug a runtime error without human help?
  NO. Zero controllability (C1-C4=0), zero observability (O1-O4=0),
  zero behavioral lift (V2=0), and zero loop completeness (V3=0).

Core behavioral gaps blocking self-debug:
  1. No structured logs (O1) — agent cannot parse app output programmatically
  2. No correlation propagation (O2) — agent cannot trace multi-component errors
  3. No config control (C1) — agent cannot fix config issues without restart
  4. No state injection (C3) — agent cannot reproduce state-dependent bugs
  5. No internal state query (O3) — agent cannot ask what state caused failure

The existing .harness/ scaffolding provides infrastructure (correlation ID
generation, log querying, health checks) but operates at the surface layer.
It wraps behavior that does not exist. The application itself must be
instrumented for any behavioral capability to be gained.

Recommended intervention: Tier 2 (Observability) — inject Type B structured
log function into `src/utils/` or `src/ink/output.ts`, activate with
`HARNESS_CONTROL=1`. This is the minimal viable step toward autonomous
self-debugging. Without it, even the OBSERVE step of the O→D→A→V loop
fails because the agent reads raw text, not structured data.
