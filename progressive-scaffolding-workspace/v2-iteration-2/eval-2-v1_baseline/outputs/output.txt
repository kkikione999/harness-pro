# Self-Debugging Assessment: Open-ClaudeCode
## Model: Sonnet | Date: 2026-04-09

---

## Executive Summary

**Can an agent self-debug a runtime error without human help?** PARTIALLY.

An agent CAN debug runtime errors when:
- Error is surfaced via tool result (`<tool_use_error>` XML)
- Error is a ShellError with captured stdout/stderr
- The fix requires file edits within allowed paths
- Permission system does not block the repair action

An agent CANNOT self-debug when:
- Error occurs in a permission-denied code path
- Error is a silent failure (no tool_use_error wrapper)
- The error requires knowledge of internal state not exposed to the agent
- Retry with different strategy would exceed permission scope

## Controllability Assessment (C1-C4)

### C1: Goal Specification — STRONG

| Dimension | Rating | Evidence |
|-----------|--------|----------|
| Goal clarity | HIGH | Query pipeline clearly routes user input through REPL -> QueryEngine -> query.ts |
| Task decomposition | HIGH | Multi-agent system via TeamCreateTool with task assignment |
| Priority handling | MEDIUM | TaskUpdateTool allows priority changes, but no explicit error-recovery task type |

**Finding:** Goal specification is well-structured. The agent receives clear tool call results and can parse error messages. However, there is no built-in "self-debug" goal type — the agent must infer debugging as a sub-goal.

### C2: Action Space — STRONG with gaps

| Tool | Capability | Gap |
|------|------------|-----|
| BashTool | Executes commands, captures stderr/stdout, exit codes | No automatic retry with different flags |
| FileEditTool | Modifies files with edit/write operations | No patch application or 3-way merge |
| GrepTool | Searches codebase patterns | No semantic search or type-aware search |
| WebFetchTool | Retrieves external documentation | No local runtime introspection |
| Task tools | Creates/updates/tracks work items | No automatic task creation on error |

**Finding:** Action space covers most debugging needs (read files, run commands, search). Missing: runtime introspection tools, automatic test execution, debugger attachment.

### C3: Constraint Enforcement — STRONG

| Mechanism | Implementation |
|-----------|----------------|
| Permission system | useCanUseTool hook chain gates all tool actions |
| 5-level settings hierarchy | user < project < local < flag < policy |
| Feature flags | feature('FLAG_NAME') prevents hardcoded checks |
| State immutability | AppStateStore requires setState() updaters |

**Finding:** Constraints are well-enforced. The permission system is the primary controllability mechanism — it prevents unauthorized actions but also limits autonomous debugging when permissions are strict.

### C4: Intervention Points — MEDIUM

| Point | Mechanism |
|-------|-----------|
| Permission dialogs | User prompted before dangerous actions |
| AbortError handling | isAbortError() distinguishes user cancel from failures |
| Plan mode | EnterPlanModeTool for structured reasoning |
| Worktree isolation | EnterWorktreeTool/ExitWorktreeTool for safe experimentation |

**Finding:** Human intervention is possible at multiple points. However, the agent cannot request temporary permission escalation for debugging — it must ask the user explicitly, breaking autonomous operation.

## Observability Assessment (O1-O4)

### O1: State Visibility — WEAK

| Aspect | Current State |
|--------|---------------|
| Tool execution | Tool results visible in conversation |
| Internal state | AppState not directly accessible to agent |
| Conversation history | useLogMessages records to transcript |
| System events | Analytics logging via logEvent() |

**Finding:** The agent sees tool results and conversation flow, but internal system state (hook states, pending operations, queue depth) is opaque. The agent cannot introspect why a tool was denied or what classifier decision was made.

### O2: Error Detection — MEDIUM

| Error Type | Detection Method |
|------------|------------------|
| Shell errors | ShellError with stdout/stderr/capture via BashTool |
| API errors | classifyAPIError() with typed tags |
| Tool validation | formatZodValidationError() |
| MCP failures | McpToolCallError class |

**Finding:** Error detection is present for tool-level failures. Missing:
- Runtime exceptions in executed code (only captured if command fails)
- Memory/performance degradation signals
- Resource exhaustion warnings
- Race condition detection

### O3: Progress Tracking — STRONG

| Mechanism | Implementation |
|-----------|----------------|
| Task system | TaskCreateTool/TaskUpdateTool with status tracking |
| Conversation compaction | useLogMessages handles large histories |
| Team coordination | MailboxProvider for inter-agent messages |
| Background tasks | LocalShellTask with foreground/background separation |

**Finding:** Progress tracking is well-implemented for long-running operations. However, there is no "step count" or "attempt count" visible to the agent for debugging loops.

### O4: Feedback Loops — WEAK

| Loop | Status |
|------|--------|
| Tool retry | No automatic retry on failure |
| Strategy adjustment | Agent must infer from error message |
| Learning from mistakes | No persistent learning across sessions |
| Classifier feedback | TRANSCRIPT_CLASSIFIER records decisions but no agent-visible learning |

**Finding:** The most significant gap. The agent receives feedback (tool errors) but has no built-in mechanism to:
1. Retry failed operations with modified parameters
2. Learn from patterns in error logs
3. Adjust behavior based on permission denials
4. Track what strategies have been attempted

## Behavioral Gap Analysis

### Gap 1: No Automatic Retry Mechanism
**Impact:** HIGH
When a bash command fails (e.g., race condition, temporary resource issue), the agent cannot automatically retry with exponential backoff or modified parameters.

### Gap 2: No Runtime State Introspection
**Impact:** HIGH
The agent cannot observe internal state (hook states, permission context, classifier decisions). When a tool is denied, the agent sees "denied" but not why.

### Gap 3: No Permission Escalation Path
**Impact:** MEDIUM
For debugging, an agent might need temporary elevated permissions. Currently, this requires explicit user approval, breaking autonomous operation.

### Gap 4: No Debugger Integration
**Impact:** MEDIUM
No LSP-based debugging tools or runtime inspection capabilities. Agent can only observe end-state, not step through execution.

### Gap 5: No Error Pattern Learning
**Impact:** MEDIUM
Each session starts fresh. The agent cannot learn from previous debugging failures or build a knowledge base of common fixes.

### Gap 6: Silent Failure Surface Area
**Impact:** LOW-MEDIUM
Some errors may not produce `<tool_use_error>` XML (e.g., network timeouts, interrupted operations). These are invisible to the agent.

## Recommendations for Enabling Self-Debugging

### Must Have (for autonomous self-debugging)
1. **Debug Hook System** — Hook that surfaces internal state to agent on error
2. **Retry Primitives** — Built-in retry mechanism with configurable backoff
3. **Error Context Exposure** — Make permission denial reasons visible to agent

### Should Have (for robust self-debugging)
4. **Debugger Tool** — LSP-based debugging integration
5. **Permission Escallation** — Temporary elevated permissions for debugging
6. **Attempt Tracking** — Visible attempt count for retry loops

### Nice to Have (for intelligent self-debugging)
7. **Pattern Memory** — Persistent storage of debugging patterns across sessions
8. **Strategy Suggestion** — Skill/agent that recommends debugging strategies
9. **Runtime Metrics** — Memory, CPU, I/O visibility for performance issues

---

## Conclusion

The Open-ClaudeCode scaffold provides a solid foundation for agent operation with strong controllability (C1-C3) and reasonable observability (O3). However, **self-debugging capability is limited by gaps in feedback loops (O4) and state visibility (O1)**.

An agent CAN debug straightforward errors (typos, missing files, syntax errors) autonomously. An agent CANNOT debug complex runtime errors that require:
- Understanding internal permission decisions
- Automatic retry without human trigger
- Runtime state introspection
- Learning from previous debugging attempts

**Key scaffolding improvement needed:** Add observability hooks that expose internal decision-making (why was this denied? what classifier ran? what retries were attempted?) to enable true autonomous debugging.
