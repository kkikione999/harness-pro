### CONTROLLABILITY (C1–C4)

--- C1: Config Control — 0/1 ---

What exists:
  The app has a 5-level settings hierarchy (user global → project local → CLI
  flag → MDM/policy). Settings are loaded at startup via getInitialSettings()
  and watched for file changes via chokidar. Tools access settings via:
    Pattern A: useSettings() hook (reactive, React)
    Pattern B: getInitialSettings() (snapshot, non-reactive)
    Pattern C: context.getAppState().settings (live within tool call)

What's missing:
  There is NO exported function that lets an external agent call
  setHarnessOverride(key, value) to mutate runtime config without restart.
  Settings come from files on disk. The only way to change config is:
    (a) Edit the settings JSON files on disk (chokidar watcher picks it up)
    (b) Pass CLI flags at startup
  Neither is a programmatic in-process function call. The AppState store is
  accessible internally via context.getAppState() but is NOT exported as a
  module that an agent can require and call.

Behavioral test (would fail):
  > HARNESS_CONTROL=1 node -e "require('./src/state').setHarnessOverride(...)"
  Result: MODULE_NOT_FOUND — no such export

Self-debug scenario — Can agent fix a runtime config error autonomously?
  NO. If a tool reads settings.timeout = 3000 and needs it to be 5000,
  the agent must: (1) know which settings file to edit, (2) edit it on disk,
  (3) wait for chokidar to detect the change, (4) hope the watcher correctly
  propagates to the right tool's context. This is file-level mutation, not
  runtime mutation. The agent cannot call a function to change behavior
  mid-session and verify it immediately.

--- C2: Feature Control — 0/1 ---

What exists:
  The app uses feature('FLAG_NAME') from bun:bundle for conditional code paths.
  Feature flags are compiled into the bundle and checked at runtime. Examples:
    feature('BASH_CLASSIFIER'), feature('TRANSCRIPT_CLASSIFIER'),
    feature('COORDINATOR_MODE'), feature('KAIROS')

What's missing:
  There is NO exported function setFeatureFlag(name, value). Feature flags are
  determined at bundle compile time by bun:bundle. The agent cannot toggle a
  feature flag programmatically.

Behavioral test (would fail):
  > HARNESS_CONTROL=1 node -e "require('./src/featureFlags').setFeatureFlag(...)"
  Result: MODULE_NOT_FOUND — no such file/module

Self-debug scenario — Can agent toggle a feature flag to test a code path?
  NO. Feature flags cannot be changed at runtime. The agent would need to
  rebuild the bundle with different bun:bundle values, which is impossible
  without the original build infrastructure.

--- C3: State Injection — 0/1 ---

What exists:

  getState()/setState() methods. State is immutable via updater functions.
  Tools can access state via ToolUseContext.getAppState().

What's missing:
  There is NO exported injectState(path, value) or queryState(path) function.
  The AppState store is internal to the app and not exported as a module.
  There are no test state hooks, no mock data injection points.

Behavioral test (would fail):
  > HARNESS_CONTROL=1 node -e "require('./src/state').injectState(...)"
  Result: MODULE_NOT_FOUND

Self-debug scenario — Can agent inject mock data to simulate failure conditions?
  NO. The agent has no way to inject test state. If debugging a tool that
  queries an external API, the agent cannot mock the response — it must
  either set up a real mock server or modify the tool's source code directly.

--- C4: Middleware Injection — 0/1 ---

What exists:
  The app has a hook system (src/utils/hooks.ts) for pre-tool-execution hooks.
  The structuredIO.ts manages SDK permission flow and MCP messaging.

What's missing:
  There is no middleware pipeline for intercepting and modifying tool requests
  or responses at runtime. The hook system is for permission/request hooks,
  not general-purpose middleware injection.

Self-debug scenario — Can agent add middleware to trace a specific request?
  NO. No middleware injection capability.

## BEHAVIORAL GAPS: SELF-DEBUG CAPABILITY
================================================================================

Central question: Can an agent self-debug a runtime error without human help?

ANSWER: NO — for the following structural reasons:

### GAP 1: No Structured Observable Output

The app emits human-readable terminal UI text via Ink. An agent running in
non-interactive mode (e.g., SDK, print mode) sees text output but cannot
parse structured log entries. When a runtime error occurs, the agent sees:

  "<error>Tool permission denied: Bash tool requires approval</error>"

Instead of a structured JSON log entry like:

  {"level":"ERROR","timestamp":"...","component":"permission",
   "message":"Tool permission denied","metadata":{"tool":"Bash","reason":"..."}}

The agent cannot OBSERVE the failure in a machine-readable way. It must
parse text, which is fragile and unreliable for autonomous operation.

### GAP 2: No Runtime Config Mutation

If the agent diagnoses that the error is caused by a misconfigured setting,
it has no way to fix it programmatically. It would need to:
  (a) Know the exact file path and format of the settings file
  (b) Edit the file on disk
  (c) Hope the chokidar watcher picks up the change correctly
There is no setHarnessOverride('toolPermissionContext.mode', 'allow') function
it can call to change behavior immediately and verify the result.

### GAP 3: No State Query

The agent cannot ask "what is the current AppState?" to understand the full
runtime context of an error. It can only infer from the visible terminal
output, which is incomplete and unstructured.

### GAP 4: No Build/Test Infrastructure

Even if harness injection patches were created, there is no way to:
  (a) Compile TypeScript source with injected patches
  (b) Run automated behavioral tests to verify the patches work
  (c) Re-bundle the CLI with the changes

The project explicitly states it is READ-ONLY research. There is no
tsconfig.json, no build script, and no test runner. The .harness/test-auto.sh
script falls through to "no test framework detected" and exits with code 1.

### GAP 5: REPL Interaction Model

The CLI's primary interface is an interactive REPL. Even if the app could
emit structured logs, an agent would need a non-interactive execution mode
to run the O→D→A→V loop programmatically. The print mode (src/cli/print.ts)
provides non-interactive output, but it still uses Ink rendering for the
terminal UI, not structured JSON for machines.

### GAP 6: Settings Are File-Based, Not Memory-Based

Unlike a typical web service that reads config at startup and can be mutated
via an API endpoint, Claude Code's settings are read from JSON files. The
AppState store is updated when the chokidar watcher detects file changes,
but this requires disk I/O and is not an atomic in-process function call.
An agent cannot "set config and verify immediately" — it must wait for the
filesystem watcher to propagate changes.

## ROOT CAUSE ANALYSIS
================================================================================

The fundamental problem is a CATEGORY MISMATCH:

  The .harness/ scaffolding is designed for a SERVICE (web server, daemon).
  Open-ClaudeCode is a TERMINAL CLI (interactive REPL, pre-compiled bundle).

v2 scaffolding expects:
  - App runs as a service (long-running process)
  - App emits structured JSON logs to stdout
  - Agent can call functions to mutate runtime state
  - Agent can parse structured logs and verify behavioral changes

Open-ClaudeCode reality:
  - App is an interactive REPL (short-lived per-session)
  - App emits terminal UI text via Ink (not machine-parseable)
  - Source cannot be rebuilt (no build infrastructure)
  - Pre-compiled bundle (package/cli.js) is immutable without rebuild

The .harness/ wrappers are INERT: they call the CLI binary and pipe its
output, but they cannot inject behavioral changes into the app because:
  1. The CLI is a pre-compiled bundle, not source
  2. Even if it were source, there's no TypeScript compiler available
  3. The CLI runs in REPL mode, not as a service with an HTTP API

Additionally, the existing assessment file incorrectly references
"src/main.tsx console.warn/error calls" — in fact, src/main.tsx is
~800k lines because it contains the entire bundled application from
source maps, not a hand-written entry point. The actual entry is
package/cli.js which is the compiled artifact.

## RECOMMENDED TIER AND PATH
================================================================================

Recommended Tier: Tier 0 — Surface (wrapper scripts only)

This is the correct tier because:
  - The app CANNOT be modified at the source level (no build infrastructure)
  - The app runs in REPL mode, not as a service with an HTTP API
  - Behavioral injection into the compiled bundle is not feasible
  - Only wrapper-level operations are possible

If Tier 1+ is desired, prerequisite work is required:

  A. Establish a build system (tsconfig, tsc compilation, bundling)
     — Without this, no source file can be compiled and tested
     — Even harness injection markers cannot be validated

  B. Enable a non-interactive execution mode that emits structured JSON
     — Current print mode still uses Ink terminal rendering
     — Agent needs machine-readable output, not terminal text

  C. Create a headless/non-REPL variant callable programmatically
     — The REPL is the primary interface; there's no headless mode
     — Agent cannot run O→D→A→V loop in interactive REPL

  D. Create a CI pipeline that runs behavioral tests against the compiled CLI
     — .harness/test-auto.sh currently exits 1 (no test framework)
     — Behavioral verification (Phase 6 of v2 REFINE LOOP) cannot run

Only after prerequisites A–D can the behavioral injection tiers be applied.

Concrete Tier 1 injection targets (once prerequisites met):
  1. src/utils/harnessConfig.ts — setHarnessOverride/getHarnessConfig
     Instrument key settings reads (getInitialSettings, useSettings)
  2. src/utils/harnessLog.ts — harnessLog() with JSON to stdout
     Replace key log calls with structured output when HARNESS_CONTROL=1
  3. src/state/harnessState.ts — injectState/queryState
     Integrate with AppStateStore for read operations
