# 测试用例：任务队列管理服务

## 背景

用于验证 harness 工具链有效性的测试候选案例。

## Prompt

```
请实现一个任务队列管理服务。

## 功能需求

1. 创建任务：提交任务（含描述、优先级），返回任务 ID
2. 任务认领：Worker 认领一个待处理任务
3. 完成任务：标记任务完成
4. 任务失败：执行失败时标记，可配置重试次数
5. 查询状态：查询任务的当前状态

## 技术要求

- SQLite 本地存储
- HTTP API
- 优先级：low, medium, high（数字越大优先级越高）
- 每个任务最多重试 3 次

## API 接口

- `POST /tasks` — 创建任务
- `GET /tasks` — 列出所有任务（支持 `?status=` 过滤）
- `GET /tasks/{id}` — 获取单个任务详情
- `POST /tasks/{id}/claim` — Worker 认领任务
- `POST /tasks/{id}/complete` — 完成任务
- `POST /tasks/{id}/fail` — 标记失败
- `GET /queues` — 查看队列状态

## 验收标准

1. 创建任务后，状态为 `pending`
2. Worker 认领后，状态变为 `running`，且其他 Worker 无法同时认领
3. 优先级高的任务先被认领
4. 任务失败后自动重试，最多 3 次
5. 完成后状态变为 `completed`
6. 不存在的任务返回 404
```

## AI 实现结果

| 指标 | 结果 |
|------|------|
| 是否一次性完成 | ✅ 是 |
| 测试数量 | 16 个 |
| 遇到的技术难点 | SQLite FOR UPDATE 不支持、重试逻辑语义、列索引 bug |
| AI 是否自行解决 | ✅ 是 |

## 评估

- **难度**：中等（有一些真实的技术挑战）
- **架构分层**：有（API Layer + Storage Layer）
- **是否需要渐进式披露**：否（AI 单次 prompt 即可完成）
- **Token 消耗**：约 100K

## 结论

该用例虽有挑战性，但 AI 能独立完成，**不适合**作为验证 harness 工具链有效性的测试用例。

需要寻找更复杂的例子。
